# STD-01: 开发质量标准

**生效日期**: 2025-11-30
**状态**: 执行中

---

## 1. 概述

本文档定义了 BobCRM 项目的开发质量标准。"完成" (Done) 不仅仅意味着代码编写完毕，还必须在易用性、美观度和可靠性上达到高标准。

## 2. 评估维度

### 2.1 功能性 (Functionality) - "做什么"
- **完整性**: 用户故事/任务中的所有需求均已实现。
- **正确性**: 功能在正常路径和边缘情况下均按预期工作。
- **集成性**: 功能在整个系统中能正确交互。

### 2.2 易用性 (Usability) - "怎么用"
- **可配置性**: 用户必须能够控制功能行为（例如：按钮动作必须是可配置的）。
- **反馈**: 系统对所有操作提供清晰的反馈（成功、错误、加载状态）。
- **效率**: 常用任务应能以最少的点击完成。
- **状态感知**: 组件应适应其上下文（例如：在详情视图中只读，在编辑视图中可编辑）。

### 2.3 美观性 (Aesthetics) - "长啥样"
- **视觉打磨**: 杜绝 "开发者审美"。正确使用 Ant Design 组件。
- **一致性**: 边距、填充、字体和颜色必须符合设计系统。
- **响应式**: 布局必须适应不同的屏幕尺寸。
- **"高级"感**: 避免使用原生浏览器弹窗 (Alert/Prompt)。使用丰富的 UI 组件。

### 2.4 可靠性 (Reliability) - "可信度"
- **错误处理**: 无静默失败。错误应被捕获并以友好的方式显示。
- **数据完整性**: 系统/自定义属性必须被正确分类和保护。
- **国际化 (I18n)**: 无硬编码字符串。所有文本必须可翻译。

## 3. 特定组件标准

### 3.1 表单设计器 (Form Designer)
- **控件**: 每个控件必须有完整的属性编辑器。
- **预览**: 设计态预览必须与运行时渲染高度一致。
- **动作**: 交互式控件（如按钮）必须具有可配置的动作。

### 3.2 系统设置 (System Settings)
- **深度**: 设置应提供细粒度的控制，而不仅仅是基本的增删改查。
- **继承**: 系统设置与用户设置之间应有清晰的区分和继承逻辑。

### 3.3 实体系统 (Entity System)
- **分类**: 系统字段与自定义字段之间应有清晰的视觉区分。
- **元数据**: 准确反映后端元数据（来源、类型等）。

## 4. "完成"检查清单
- [ ] 功能需求已满足？
- [ ] 易用性已验证（用户真的能用吗）？
- [ ] 美观性已检查（看起来好吗）？
- [ ] 可靠性已测试（错误处理、边缘情况）？
- [ ] 国际化已完成（无硬编码）？
